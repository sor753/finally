# FinAlly — AI Trading Workstation

## 1. ビジョン

FinAlly（Finance Ally）は、視覚的に美しいAI搭載のトレーディングワークステーションである。
リアルタイムの市場データをストリーミング配信し、ユーザーはシミュレーションポートフォリオで取引を行うことができ、さらにポジション分析やユーザーに代わって取引を実行できるLLMチャットアシスタントを統合する。まるでAIコパイロットを搭載した最新のBloomberg端末のような外観と操作感を備える。

これは、エージェント型AIコーディングコースの集大成となるプロジェクトである。AIエージェントが連携して、本番環境レベルのフルスタックアプリケーションを構築できることを示すために、コーディングエージェントによって完全に構築されている。エージェントは`planning/`ディレクトリ内のファイルを介して相互作用する。

## 2. ユーザーエクスペリエンス

### 初回起動

ユーザーはDockerコマンド（または提供されている起動スクリプト）を1つ実行すると、ブラウザが`http://localhost:8000`を開く。ログインもサインアップも不要で、すぐに以下の画面が表示される。

- リアルタイムで価格が更新されるグリッド形式の、デフォルトのティッカーシンボル10個のウォッチリスト
- 仮想資金10,000ドル
- ダークトーンでデータ豊富なトレーディングターミナルのデザイン
- すぐにサポートを受けられるAIチャットパネル

### ユーザーができること

- **価格ストリームの監視** — 価格は上昇時は緑色、下落時は赤色で点滅し、CSSアニメーションで徐々にフェードアウトする。
- **スパークラインミニチャートの表示** — ウォッチリストの各銘柄の横に価格変動を表示する（実装詳細はセクション10参照）。
- **銘柄をクリック**すると、メインチャートエリアに詳細なチャートが表示される。
- **株式の売買** — 成行注文のみ、現在の価格で即時約定、手数料無料、確認ダイアログなし
- **ポートフォリオの監視** — ポジションをウェイト別にサイズ表示し、損益に応じて色分けしたヒートマップ（ツリーマップ）と、ポートフォリオ全体の価値を時系列で追跡する損益チャートを表示する。
- **ポジション一覧表の表示** — 銘柄、数量、平均取得価格、現在の価格、未実現損益、変動率を表示する。
- **AIアシスタントとのチャット** — ポートフォリオに関する質問、分析結果の取得、AIによる取引実行、ウォッチリストの管理を自然な会話で行える。
- **ウォッチリストの管理** — 銘柄の追加/削除を手動で行うか、AIチャット経由で行う。

### ビジュアルデザイン

- **ダークテーマ**: 背景色は`#0d1117`または`#1a1a2e`付近、境界線は控えめなグレー、純粋な黒は使用しない。
- **価格フラッシュアニメーション**: 価格変動時に背景が緑/赤で短時間ハイライト表示され、CSSトランジションにより約500msかけてフェードアウトする。
- **接続状態インジケーター**: ヘッダーに小さな色のドットを表示する（緑＝接続済み、黄＝再接続中、赤＝切断済み）。
- **プロフェッショナルでデータ密度の高いレイアウト**: Bloombergやトレーディング端末にインスパイアされた、すべてのピクセルが意味を持つレイアウトを採用する。
- **レスポンシブデザインでありながらデスクトップファースト**: ワイドスクリーンに最適化され、タブレットでも快適に動作する。

## 3. アーキテクチャ概要

### 単一コンテナ、単一ポート

```
┌─────────────────────────────────────────────────┐
│  Docker Container (port 8000)                   │
│                                                 │
│  FastAPI (Python/uv)                            │
│  ├── /api/*          REST endpoints             │
│  ├── /api/stream/*   SSE streaming              │
│  └── /*              Static file serving         │
│                      (Next.js export)            │
│                                                 │
│  SQLite database (volume-mounted)               │
│  Background task: market data polling/sim        │
└─────────────────────────────────────────────────┘
```

各コンポーネントの詳細（ビルド手順・依存関係・設定）は実装時にセクション11を再確認すること。

### なぜこれらの選択をするのか

| Decision                  | Rationale                                                                                    |
| ------------------------- | -------------------------------------------------------------------------------------------- |
| SSE（WebSocket の代わり） | 一方向プッシュのみで十分。シンプルで双方向の複雑さがなく、あらゆるブラウザに対応             |
| 静的 Next.js エクスポート | シングルオリジン、CORS の問題なし、ポート 1 つ、コンテナ 1 つ、シンプルなデプロイメント      |
| SQLite（PostgreSQL の代わり） | 認証なし = マルチユーザーなし = データベース サーバー不要。自己完結型、設定不要          |
| 単一の Docker コンテナ    | コマンド 1 つを実行するだけ。本番環境で docker-compose やサービス オーケストレーションは不要 |
| Python 用 uv              | 高速で最新の Python プロジェクト管理。再現可能なロックファイル。学ぶべき内容                 |
| 成行注文のみ              | オーダーブック、指値注文ロジック、部分約定を排除 - ポートフォリオ計算が劇的に簡素化          |

## 5. 環境変数

```bash
# 必須：LLMチャット機能用のOpenRouter APIキー
OPENROUTER_API_KEY=your-openrouter-api-key-here

# オプション：実際の市場データを取得するために、Massive（Polygon.io）APIキーを指定する。
# 設定しない場合は、内蔵の市場シミュレーターが使用される（ほとんどのユーザーに推奨）。
MASSIVE_API_KEY=

# オプション: Massive API有料ティアのポーリング間隔（秒）。未設定時は5秒。2〜15秒を推奨。
MASSIVE_POLL_INTERVAL_SECONDS=

# オプション: 決定論的なモック LLM 応答（テスト用）の場合は「true」に設定
LLM_MOCK=false
```

### 動作

- `MASSIVE_API_KEY` が設定されていて空でない場合 → バックエンドは市場データ取得に Massive REST API を使用する。
- `MASSIVE_API_KEY` が設定されていないか空の場合 → バックエンドは組み込みの市場シミュレーターを使用する。
- `LLM_MOCK=true` の場合 → バックエンドは以下の固定JSONをLLMレスポンスとして返す（E2Eテストのアサーション基準）。

```json
{
  "message": "FinAllyモードです。AAPLを1株購入しました。",
  "trades": [{ "ticker": "AAPL", "side": "buy", "quantity": 1 }],
  "watchlist_changes": []
}
```
- バックエンドはプロジェクトルートから `.env` ファイルを読み込む（コンテナにマウントされるか、docker `--env-file` で読み込まれる）。

## 6. 市場データ

### 2つの実装、1つのインターフェース

シミュレータとMassiveクライアントは、同じ抽象インターフェースを実装する。バックエンドは環境変数に基づいてどちらを使用するかを選択する。下流のすべてのコード（SSEストリーミング、価格キャッシュ、フロントエンド）は、ソースに依存しない。

### シミュレーター（デフォルト）

- 幾何ブラウン運動（GBM）を用いて価格を生成し、銘柄ごとにドリフトとボラティリティを設定できる。
- 約500ミリ秒間隔で更新する。
- 銘柄間で相関した値動きを再現する（例: ハイテク株は連動して動く）。
- ドラマチックな演出として、銘柄が突然2〜5%変動するランダムな「イベント」を時折発生させる。
- 現実的な初期価格（例: AAPL 約190ドル、GOOGL 約175ドルなど）から開始する。
- プロセス内バックグラウンドタスクとして実行され、外部依存関係はない。

### Massive API（オプション）

- REST API ポーリングを使用する（WebSocket ではない）。よりシンプルで、すべてのティアで動作する。
- 設定可能な間隔で、監視対象のすべての銘柄の統合値をポーリングする。
- 無料ティア（5回/分）: 15秒ごとにポーリングする。
- 有料ティア: デフォルト5秒ごとにポーリングする。`MASSIVE_POLL_INTERVAL_SECONDS` 環境変数で上書き可能（2〜15秒を推奨）。
- REST レスポンスを解析し、シミュレーターと同じ形式に変換する。

### ティッカーの扱い

- ウォッチリストおよび取引では、デフォルトの10銘柄に限らず任意のティッカーを許可する。
- シミュレーターは未知の銘柄に対し、デフォルトパラメータ（初期価格 50〜200ドルの範囲でランダム・標準ボラティリティ）で価格系列を動的に生成する。シミュレーターモードでは未知ティッカーに対して `404` は返さない。
- Massive API 利用時は、API が認識するティッカーであれば利用できる。API が認識しないティッカーを取引またはウォッチリストへ追加しようとした場合のみ `404` を返す。
- ウォッチリストの上限は50件とし、超過時は `POST /api/watchlist` が `422` を返す。

### 共有価格キャッシュ

- 単一のバックグラウンドタスク（シミュレーターまたはMassiveポーラー）がインメモリ価格キャッシュに書き込む。
- キャッシュには、各銘柄の最新価格、前価格、およびタイムスタンプが格納される。
- SSEストリームはこのキャッシュからデータを読み取り、接続されているクライアントに更新情報をプッシュする。
- このアーキテクチャは、データレイヤーを変更することなく、将来のマルチユーザーシナリオをサポートする。

### SSEストリーミング

- エンドポイント：`GET /api/stream/prices`
- 長時間のSSE接続。クライアントはネイティブの`EventSource` APIを使用する。
- サーバーは、システムが認識しているすべての銘柄の価格更新情報を一定の間隔（約500ms）でプッシュする。配信対象銘柄の範囲（ウォッチリストのみか、保有ポジション銘柄も含むかなど）は実装時に要検討。
> **既知の制約**: 全銘柄を全クライアントに一律配信する設計はシングルユーザー前提。マルチユーザー化の際はクライアント別フィルタリングが必要。
- 各SSEイベントは、システムが認識しているすべての銘柄を1イベントにまとめ、配列で送信する。
- クライアントは再接続を自動的に処理する（EventSourceにはリトライ機能が組み込まれている）。

各イベントの `data` フィールドのJSON形状は以下のとおり。`direction` は `"up" | "down" | "flat"` のいずれか。

```json
{
  "type": "prices",
  "timestamp": "2026-05-22T09:30:00.000Z",
  "prices": [
    { "ticker": "AAPL", "price": 190.12, "prev_price": 189.80, "direction": "up" }
  ]
}
```

## 7. データベース

### SQLiteの遅延初期化

バックエンドは起動時（または最初のリクエスト時）にSQLiteデータベースの存在を確認する。ファイルが存在しない場合、またはテーブルが不足している場合は、スキーマを作成し、デフォルトデータをシードする。これにより、以下のメリットが得られる。

- 別途移行手順は不要
- 手動でのデータベース設定は不要
- 新規のDockerボリュームは、自動的にクリーンなシード済みデータベースで起動

> **既知の制約**: 初回リクエスト時にスキーマ作成とシードが実行されるため、わずかな遅延が発生する。デモ用途では許容範囲とし、起動後の最初のリクエストで初期化が完了することを前提とする。

### スキーマ

すべてのテーブルには、デフォルト値が「default」の`user_id`列が含まれている。これは現状（シングルユーザー）ではハードコーディングされるが、将来的にスキーマ移行なしでマルチユーザーをサポートすることを可能にする。

> **既知の制約**: `REAL` 型は IEEE 754 浮動小数点であり、金融計算で丸め誤差が生じる場合がある。本アプリはシミュレーション用途のため許容するが、実装時は累積誤差に注意すること。

**user_profile** — ユーザーの状態（キャッシュバランス）

- `id` TEXT PRIMARY KEY (default: `"default"`)
- `cash_balance` REAL (default: `10000.0`)
- `created_at` TEXT (ISO timestamp)

**watchlist** — ユーザーが監視している銘柄

- `id` TEXT PRIMARY KEY (UUID)
- `user_id` TEXT (default: `"default"`)
- `ticker` TEXT
- `added_at` TEXT (ISO timestamp)
- `(user_id, ticker)` に一意性制約あり

**positions** — 現在の保有銘柄（ユーザーごとに銘柄ごとに1行）

- `id` TEXT PRIMARY KEY (UUID)
- `user_id` TEXT (default: `"default"`)
- `ticker` TEXT
- `quantity` REAL (端数株対応)
- `avg_cost` REAL
- `updated_at` TEXT (ISO timestamp)
- `(user_id, ticker)` に一意性制約あり

**trades** — 取引履歴 (追記専用ログ)

- `id` TEXT PRIMARY KEY (UUID)
- `user_id` TEXT (default: `"default"`)
- `ticker` TEXT
- `side` TEXT (`"buy"` または `"sell"`)
- `quantity` REAL (端数株対応)
- `price` REAL
- `executed_at` TEXT (ISO timestamp)

**portfolio_snapshots** — ポートフォリオの時系列価値 (損益グラフ用)。市場データタスクとは独立した専用のバックグラウンドタスクにより30秒ごとに、また各取引実行直後に記録される。`/api/portfolio/history` は初期は全件を返し、データ点が増えて問題になった段階で等間隔ダウンサンプリング（最大200点を目安）を追加する。

- `id` TEXT PRIMARY KEY (UUID)
- `user_id` TEXT (default: `"default"`)
- `total_value` REAL
- `recorded_at` TEXT (ISO timestamp)

**chat_messages** — LLMとの会話履歴

- `id` TEXT PRIMARY KEY (UUID)
- `user_id` TEXT (default: `"default"`)
- `role` TEXT (`"user"` または `"assistant"`)
- `content` TEXT
- `actions` TEXT (JSON — 実行された取引、ウォッチリストの変更。ユーザーメッセージの場合はnull)
- `created_at` TEXT (ISO timestamp)

### 初期シードデータ

- ユーザープロファイル1つ: `id="default"`、`cash_balance=10000.0`
- ウォッチリストエントリ10件: AAPL、GOOGL、MSFT、AMZN、TSLA、NVDA、META、JPM、V、NFLX

## 8. API Endpoints

### Market Data

| Method | Path                 | Description               |
| ------ | -------------------- | ------------------------- |
| GET    | `/api/stream/prices` | 価格最新情報の SSE stream |

### Portfolio

| Method | Path                     | Description                                                    |
| ------ | ------------------------ | -------------------------------------------------------------- |
| GET    | `/api/portfolio`         | 現在のポジション、現金残高、総資産額、未実現損益               |
| POST   | `/api/portfolio/trade`   | 取引を実行する: `{ticker, quantity, side}`                     |
| GET    | `/api/portfolio/history` | ポートフォリオ価値の推移を示すスナップショット（損益グラフ用） |

#### `GET /api/portfolio` レスポンス

```json
{
  "cash_balance": 8500.0,
  "total_value": 12400.0,
  "positions": [
    {
      "ticker": "AAPL",
      "quantity": 10,
      "avg_cost": 185.0,
      "current_price": 190.0,
      "unrealized_pnl": 50.0,
      "pnl_pct": 2.7
    }
  ]
}
```

`total_value` は現金 + 全保有銘柄の時価評価額の合計。

#### `GET /api/portfolio/history` レスポンス

```json
{
  "snapshots": [
    { "total_value": 10000.0, "recorded_at": "2026-05-22T09:30:00.000Z" },
    { "total_value": 10243.5, "recorded_at": "2026-05-22T09:30:30.000Z" }
  ]
}
```

`snapshots` は `recorded_at` 昇順。データ点が多い場合は最大200点に等間隔ダウンサンプリングして返す。

#### `POST /api/portfolio/trade` レスポンス

```json
{
  "ticker": "AAPL",
  "side": "buy",
  "quantity": 10,
  "price": 190.12,
  "cash_balance": 8098.80
}
```

### Watchlist

| Method | Path                      | Description                      |
| ------ | ------------------------- | -------------------------------- |
| GET    | `/api/watchlist`          | 最新価格付きのウォッチリスト銘柄（価格キャッシュが空の場合は `price: null`） |
| POST   | `/api/watchlist`          | 銘柄コード追加: `{ticker}`       |
| DELETE | `/api/watchlist/{ticker}` | 銘柄コード削除                   |

#### `GET /api/watchlist` レスポンス

```json
{
  "watchlist": [
    { "ticker": "AAPL", "price": 190.12, "added_at": "2026-05-22T09:30:00.000Z" }
  ]
}
```

#### `POST /api/watchlist` レスポンス（201 Created）

```json
{ "ticker": "PYPL", "added_at": "2026-05-22T09:30:00.000Z" }
```

#### `DELETE /api/watchlist/{ticker}` レスポンス

204 No Content（ボディなし）

### Chat

| Method | Path        | Description                                                                            |
| ------ | ----------- | -------------------------------------------------------------------------------------- |
| POST   | `/api/chat` | メッセージを送信すると、完全なJSONレスポンス(message + executed actions)が返される。 |

#### `POST /api/chat` レスポンス

```json
{
  "message": "AAPLを10株購入しました。",
  "executed_trades": [{ "ticker": "AAPL", "side": "buy", "quantity": 10, "price": 190.12 }],
  "watchlist_changes": [{ "ticker": "PYPL", "action": "add" }],
  "errors": []
}
```

`executed_trades` はLLMが指定して実際に約定した取引（価格付き）。`errors` は検証失敗メッセージの配列。

### System

| Method | Path          | Description                          |
| ------ | ------------- | ------------------------------------ |
| GET    | `/api/health` | Health check (for Docker/deployment) |

### エラーレスポンス

- エラーは FastAPI 標準形式 `{"detail": "<メッセージ>"}` で返す。
- HTTP ステータス: 取引検証失敗（資金不足・株数不足）は `422`、リクエスト本文の不備は `422`。未知ティッカーの扱いはモード依存（シミュレーターモードでは任意ティッカーを動的生成するため `404` は返さない。Massive API モードでは API が認識しないティッカーのみ `404` を返す）。
- フロントエンドは `detail` の文字列をそのまま表示する。

## 9. LLM統合

LLMを呼び出すコードを記述する際は、cerebras-inferenceスキルを使用して、OpenRouter経由でLiteLLMを`openrouter/openai/gpt-oss-120b`モデルに渡して、推論プロバイダとしてCerebrasを使用する。結果の解釈には構造化出力を使用する必要がある。

プロジェクトルートの.envファイルにOPENROUTER_API_KEYを記載する。

### 仕組み

ユーザーがチャットメッセージを送信すると、バックエンドは以下の処理を実行する。

1. ユーザーの現在のポートフォリオコンテキスト（現金、損益のあるポジション、リアルタイム価格のウォッチリスト、ポートフォリオの合計価値）を読み込む。
2. `chat_messages` テーブルから直近20件（ユーザー/アシスタント10往復相当）の会話履歴を読み込む。
3. システムメッセージ、ポートフォリオコンテキスト、会話履歴、およびユーザーの新しいメッセージを含むプロンプトを作成する。
4. cerebras-inference スキルを使用して、LiteLLM → OpenRouter 経由で LLM を呼び出し、構造化出力を要求する。
5. 構造化された JSON レスポンス全体を解析する。
6. レスポンスの `trades` フィールドで指定された各取引を実行し、約定時の実際の価格を付加して `executed_trades` リストを構築する。ウォッチリストの変更も同様に実行する。
7. メッセージと実行されたアクションを `chat_messages` テーブルに保存する。保存前に `message` フィールドの内容をサニタイズし、スクリプトインジェクション等の悪意あるコンテンツを除去する。
8. フロントエンドに完全な JSON レスポンスを返す。LLM 出力の `trades`（価格なし）はクライアントには返さず、約定済みの `executed_trades`（価格付き）のみを返す（トークンごとのストリーミングは行わない。Cerebras の推論は十分高速であるため、読み込みインジケーターで十分）。

### 構造化スキーマ出力

LLMは、以下のスキーマに一致するJSONで応答するように指示される:

```json
{
  "message": "ユーザーへの会話応答",
  "trades": [{ "ticker": "AAPL", "side": "buy", "quantity": 10 }],
  "watchlist_changes": [{ "ticker": "PYPL", "action": "add" }]
}
```

- `message` (必須): ユーザーに表示される会話テキスト
- `trades` (オプション): 自動実行する取引の配列。各取引は、手動取引と同様の検証（買いの場合は十分な現金、売りの場合は十分な株数）を受ける。欠落時は空配列として扱う。
- `watchlist_changes` (オプション): ウォッチリストの変更内容の配列。欠落時は空配列として扱う。

### 自動実行

LLMが指定した取引は、確認ダイアログなしで自動的に実行される。これは意図的な設計である。

- 仮想資金を使用したシミュレーション環境なので、リスクはゼロ。
- スムーズで印象的なデモ体験を提供する。
- コースの中核テーマである、エージェント型AIの機能を実証する。

取引の検証に失敗した場合（例：資金不足）、エラーメッセージはチャット応答に含まれ、LLMがユーザーに通知される。

### システムプロンプトガイダンス

LLMには「FinAlly、AIトレーディングアシスタント」というメッセージが表示され、以下の指示が与えられる。

- ポートフォリオ構成、リスク集中度、損益を分析する
- 根拠に基づいたトレード提案を行う
- ユーザーの要求または同意に基づいてトレードを実行する
- ウォッチリストを積極的に管理する
- 簡潔かつデータに基づいた応答を行う
- 常に有効な構造化JSONで応答する

### LLMモックモード

`LLM_MOCK=true` の場合の動作はセクション5参照。

## 10. フロントエンドデザイン

### レイアウト

フロントエンドは、ターミナル風の高密度レイアウトを採用したシングルページアプリケーションである。具体的なコンポーネントアーキテクチャとレイアウトシステムはフロントエンドエンジニアに委ねられるが、UIには以下の要素を含める必要がある。

- **ウォッチリストパネル** — ウォッチリスト対象の銘柄をグリッド/テーブル形式で表示する。銘柄コード、現在の価格（変動時に緑/赤で点滅）、直近変動率（SSEの `prev_price` との差分）、スパークラインミニチャート（ページ読み込み以降のSSEからの累積値、銘柄ごと最新500点上限）を表示する。最大50銘柄 × 500点のメモリ使用量はデモ規模では許容範囲とする。
- **メインチャートエリア** — 現在選択されている銘柄の価格推移を折れ線チャートで表示する（データソースはスパークラインと同じく、ページ読み込み以降のSSE累積価格）。ウォッチリストで銘柄をクリックすると、このチャートでその銘柄が選択される。
- **ポートフォリオヒートマップ** — 各長方形がポジションを表し、ポートフォリオのウェイトに応じてサイズを調整し、損益（緑＝利益、赤＝損失）に応じて色分けしたツリーマップ形式で可視化する。
- **損益チャート** — `portfolio_snapshots` のデータを使用し、ポートフォリオの総額の推移を折れ線グラフで示す。
- **ポジション一覧表** — すべてのポジションを表形式で一覧表示する（ティッカー、数量、平均取得価格、現在価格、未実現損益、変動率）。
- **トレードバー** — シンプルな入力エリア（ティッカー、数量、買いボタン、売りボタン）。成行注文として即時約定する。
- **AIチャットパネル** — ドッキング/折りたたみ可能なサイドバー。メッセージ入力、スクロール可能な会話履歴、LLMからの応答待ち中の読み込みインジケーターを備える。取引の約定とウォッチリストの変更は確認としてインラインで表示される。
- **ヘッダー** — ポートフォリオの合計価値（リアルタイム更新）、接続ステータス表示、現金残高を表示する。

### 技術情報

- `/api/stream/prices` への SSE 接続には `EventSource` を使用する。
- チャートライブラリは **Lightweight Charts** を使用する（Canvasベース・高パフォーマンス・スパークライン対応）。
- 価格フラッシュ効果: 新しい価格を受信した際に、背景色トランジション付きの CSS クラスを短時間適用し、その後削除する。
- すべての API 呼び出しは同一のオリジン (`/api/*`) に送信されるため、CORS 設定は不要である。
- カスタムダークテーマのスタイリングには Tailwind CSS を使用する。

## 11. Docker & デプロイメント

### Multi-Stage Dockerfile

```
Stage 1: Node 20 slim
  - Copy frontend/
  - npm ci && npm run build (produces static export)

Stage 2: Python 3.12 slim
  - Install uv
  - Copy backend/
  - uv sync (install Python dependencies from lockfile)
  - Copy frontend build output into a static/ directory
  - Expose port 8000
  - CMD: uvicorn serving FastAPI app
```

FastAPIは、静的フロントエンドファイルとすべてのAPIルートをポート8000で提供する。

### Dockerボリューム

SQLiteデータベースは、名前付きDockerボリュームを介して永続化される。

```bash
docker run -v finally-data:/app/db -p 8000:8000 --env-file .env finally
```

プロジェクトルートの`db/`ディレクトリは、コンテナ内の`/app/db`にマッピングされる。バックエンドはこのパスに`finally.db`というファイルを書き込む。

### Start/Stop Scripts

`start_*` と `stop_*` の各スクリプトは macOS/Linux（`.sh`）と Windows PowerShell（`.ps1`）の2種類があり、動作は共通。

**start スクリプト**: 未ビルド時（または強制ビルドフラグ指定時）にイメージをビルドし、ボリュームマウント・ポートマッピング・`.env`ファイルを指定してコンテナを起動。URLを表示し、必要に応じてブラウザを開く。強制ビルドフラグは macOS が `--build`、Windows が `-Build`。

**stop スクリプト**: 実行中のコンテナを停止・削除する。ボリュームは保持。

すべてのスクリプトは冪等性を持つ。つまり、複数回実行しても安全であるべきである。

### オプションのクラウドデプロイ

このコンテナは、AWS App Runner、Render、またはその他のコンテナプラットフォームへのデプロイを想定して設計されている。App Runner用のTerraform構成は、ストレッチゴールとして`deploy/`ディレクトリに提供される可能性があるが、コアビルドには含まれない。

## 12. テスト戦略

### 単体テスト (`frontend/` および `backend/` 内)

**バックエンド (pytest)**:

- マーケットデータ: シミュレーターが有効な価格を生成、GBM 計算が正しい、Massive API レスポンスの解析が機能、両方の実装が抽象インターフェースに準拠
- ポートフォリオ: 取引実行ロジック、損益計算、エッジケース (保有量以上の売却、不足資金での購入、損失での売却)
- LLM: 構造化出力解析がすべての有効なスキーマを処理、不正なレスポンスの適切な処理、チャットフロー内での取引検証
- API ルート: 正しいステータスコード、レスポンス形式、エラー処理

**フロントエンド (React Testing Library または類似ライブラリ)**:

- モックデータを使用したコンポーネントのレンダリング
- 価格変動時に価格フラッシュアニメーションが正しくトリガーされる
- ウォッチリストの CRUD 操作
- ポートフォリオ表示の計算
- チャットメッセージのレンダリングと読み込み状態

### E2E テスト ( `test/`)

**インフラストラクチャ**: `test/` ディレクトリに、アプリケーションコンテナとPlaywrightコンテナを起動する`docker-compose.test.yml`ファイルを用意する。これにより、ブラウザの依存関係を本番環境イメージから排除できる。

**環境**: テストは、速度と決定性を確保するため、デフォルトで`LLM_MOCK=true`を有効にして実行される。

**主要シナリオ**:

- 新規開始: デフォルトのウォッチリストが表示され、残高1万ドルが表示され、価格がストリーミングされる。
- ウォッチリストへの銘柄の追加と削除
- 株式の購入: 現金が減少し、ポジションが表示され、ポートフォリオが更新される。
- 株式の売却: 現金が増加し、ポジションが更新または消滅する。
- ポートフォリオの可視化: ヒートマップが正しい色でレンダリングされ、損益チャートにデータポイントが表示される。
- AIチャット（モック）：メッセージを送信し、応答を受信し、取引の実行結果がインラインで表示される。
- SSEの耐障害性: 切断と再接続の確認
