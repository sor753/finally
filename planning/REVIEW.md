## 未解決の疑問点・懸念事項

1. **SSE配信対象の銘柄範囲**: 「システムが認識しているすべての銘柄」とは具体的に何を指すか。ウォッチリスト銘柄のみ、または過去に取引された銘柄も含むか。ポジションを保有しているがウォッチリストから削除した銘柄は価格更新を受け取れるか。

2. **モックレスポンスの仕様**: `LLM_MOCK=true` 時に返される固定JSONの内容が未定義のため、E2Eテストのアサーション設計が困難である。モックレスポンスに特定の銘柄への取引指示が含まれていないと、取引フローのE2Eテストシナリオが成立しない。

3. **`portfolio_snapshots` の初回スナップショット**: アプリ起動直後（取引前）にスナップショットが存在しない場合、`GET /api/portfolio/history` は空配列を返すか。損益グラフのUIが空配列をどう扱うべきかを定義する必要がある。

4. **ウォッチリスト価格の取得タイミング**: `GET /api/watchlist` が返す `price: null` は起動直後の状態と定義されているが、SSEが接続されてからどの程度の時間で価格が確定するか。フロントエンドが `null` を受け取った場合に「読み込み中」表示をする必要があるか。

5. **Massive API無料ティアの自動検出**: `MASSIVE_POLL_INTERVAL_SECONDS` の値によって無料/有料ティアを切り分けるのか、それともそのような判断ロジックはなく、設定値がそのままポーリング間隔になるのか。無料ティアのユーザーが誤って短い間隔を設定してAPI制限に達した場合の挙動は考慮されているか。

## GPT-4o セカンドオピニオン（2026-05-26）

> GitHub Models API (GPT-4o) による自動レビュー結果。Claude の生成物に対するセカンドオピニオン。

### 重大度: 高

1. **SQLite の `REAL` 型で金融精度が失われる可能性**（セクション7）
   — `quantity` / `price` を IEEE 754 浮動小数点で保存すると丸め誤差が生じる。SQLite に `DECIMAL` 型はないため、整数化（セント単位）または `TEXT` 保存を検討すべき。

2. **LLM 出力のサニタイズが未定義**（セクション9）
   — `chat_messages` に保存する前に悪意あるコンテンツをサニタイズする処理の言及がない。

3. **構造化出力の欠落時のデフォルト動作が未定義**（セクション9）
   — `trades` / `watchlist_changes` が LLM レスポンスから欠落した場合の挙動（空配列扱いか、エラーか）が明記されていない。

### 重大度: 中

4. **SSE のスケーラビリティ**（セクション6）
   — 全銘柄を全クライアントに 500ms 毎に送る設計はクライアント数増加時に問題になる。シングルユーザー前提の現設計では許容範囲だが、将来的な拡張時に要再検討。

5. **スパークライン メモリ使用量が未試算**（セクション10）
   — 最大 50 銘柄 × 500 点のデータ保持量についての言及がない。

6. **SQLite 遅延初期化による初回リクエスト遅延**（セクション7）
   — 初回リクエスト時にスキーマ作成とシードが走るため、遅延が発生する可能性がある。

### 対応履歴（2026-05-26 反映済み）

- **1（REAL 型精度）**: セクション7に `既知の制約` 注記を追加。シミュレーション用途として許容することを明記。
- **2（サニタイズ）**: セクション9の手順7に保存前サニタイズの記述を追加。
- **3（欠落時のデフォルト）**: セクション9の `trades` / `watchlist_changes` の説明に「欠落時は空配列として扱う」を追記。
- **4（SSE スケーラビリティ）**: セクション6に `既知の制約` 注記を追加。シングルユーザー前提であることを明記。
- **5（スパークライン メモリ）**: セクション10に最大50銘柄 × 500点はデモ規模では許容範囲と明記。
- **6（遅延初期化の遅延）**: セクション7に `既知の制約` 注記を追加。初回リクエストで初期化が完了することを前提として許容することを明記。

### 全体的な設計

1. **シングルユーザー設計の制約**（セクション3・7）
   — `user_id` が固定値 `"default"` になっている点は、マルチユーザー化の際に問題となる可能性がある。設計上は `user_id` カラムを全テーブルに組み込んでおり将来対応の準備はできているが、APIレイヤーでの認証・セッション管理についての言及がない。

2. **SSEのスケーラビリティ**（セクション6）
   — 全クライアントに全銘柄の価格情報を送信する設計は、クライアント数増加時にネットワーク帯域を圧迫する可能性がある。クライアントごとのウォッチリストに基づくフィルタリングの導入を将来的に推奨する。（既知の制約として PLAN.md に明記済み）

3. **金融計算における精度**（セクション7）
   — SQLite の `REAL` 型は IEEE 754 浮動小数点であり、金融計算で丸め誤差が生じる場合がある。（既知の制約として PLAN.md に明記済み。シミュレーション用途として許容）

4. **エラーハンドリングの構造化**（セクション8）
   — `{"detail": "<メッセージ>"}` 形式は FastAPI 標準として統一されているが、エラーコードやエラータイプを含む構造化レスポンスにすることで、フロントエンドでの条件分岐処理がより容易になる。現状の設計では `detail` の文字列をそのまま表示する前提となっており、将来的な多言語対応やエラー種別ごとのUI分岐が困難になる可能性がある。

### セキュリティ

5. **LLM レスポンスのサニタイズ実装詳細**（セクション9）
   — セクション9の手順7に「保存前にサニタイズする」旨が記載されているが、具体的なライブラリや手法が未指定である。`bleach` などのライブラリ名を例示すると実装者の判断基準が明確になる。

6. **環境変数の管理方針**（セクション5）
   — `.env` ファイルを gitignore する設計は適切だが、クラウドデプロイ時のシークレット管理（AWS Secrets Manager、HashiCorp Vault 等）についての言及がない。セクション11の「オプションのクラウドデプロイ」箇所にシークレット管理の注意事項を1行追記することを推奨する。

7. **API エンドポイントの認証**（セクション8）
   — シングルユーザー設計でも、コンテナをクラウドに公開する場合は外部からの不正アクセスが可能となる。少なくともクラウドデプロイ時の注意点として、APIキーやIPアドレス制限などの保護手段への言及を推奨する。

### パフォーマンス

8. **価格キャッシュの有効期限**（セクション6）
   — インメモリ価格キャッシュのTTL（Time-To-Live）が未定義である。銘柄がウォッチリストから削除された後もキャッシュが無制限に保持されると、長期稼働時にメモリ使用量が増大する可能性がある。削除済み銘柄のキャッシュをどう扱うかの方針を明記することを推奨する。

9. **初回リクエスト時の遅延**（セクション7）
   — SQLite 遅延初期化により初回リクエスト時に遅延が発生する設計について、バックエンド起動時（`lifespan` イベントなど）にスキーマ作成・シードを実行するよう変更するとユーザー体験が向上する。（既知の制約として PLAN.md に明記済み）

### その他

10. **テストの異常系カバレッジ**（セクション12）
    — E2E テストシナリオは基本的なハッピーパスを網羅しているが、異常系（無効なティッカーコード、APIキーの欠如、SSE切断時の再接続シーケンス、LLMがスキーマ外のJSONを返した場合）のシナリオが不足している。これらのケースを追加することでより堅牢なテスト戦略になる。（SSE耐障害性テストは記載済み）

11. **Dockerfileビルドキャッシュ最適化**（セクション11）
    — Multi-Stage Dockerfile では `package.json` と `package-lock.json` を先にコピーしてから `npm ci` を実行することで、依存関係が変更されない場合のビルドキャッシュを最大限活用できる。現状の記述は「`npm ci && npm run build`」のみで、コピー手順の詳細が省略されている。
